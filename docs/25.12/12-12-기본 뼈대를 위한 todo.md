### 1. 역할 정리 
- TodoPage.jsx
    - 투두 상태(`todos`)와 입력 상태(`inputValue`) 관리
    - add / delete / toggle 로직 담당
    - LocalStorage 저장 / 로드(`useEffect`) 담당
    - 화면 렌더링 담당

- TodoInput.jsx
    - 체크박스 UI 컴포넌트
    - `props`로 `item`, `onToggle` 받아서 완료 토굴 이벤트 호출

### 2. 기능 명세 (현제 기준)
- Todo 추가(Create)
    - 입력값이 공백이면 추가하지 않음(`trim()` 으로 검증)
    - `id: Date.now()` 기반으로 고유값 생성
    - 추가 후 입력창 초기화

- Todo 목록 조회(Read)
    - `todos.map()`으로 리스트 렌더링
    - 각 항목에 `key={item.id}`부여

- Todo 완료 토굴(Update)
    - 체크박스 클릭 시 `completed`반전
    - 완료 상태면 텍스트에 취소선 스타일 적용

- Todo 삭제(Delete)
    - 삭제 버튼 클릭 시 해당 `id` 제거 (`filter`)

- LocalStorage 영속화
    - `todos` 변경 시 저장: 
        - `localStorage.setItem('todos', JSON.stringify(todos))`
    - 최초 진입 시 로드:
        - `JSON.parse(localStorage.getItem('todos'))`

### 3. 화면 구조 / 컴포넌트 관계
- UI 구조(개념)
``` css
TodoPage
 ├─ Input(text) + Add Button
 └─ Todo List (ul)
     └─ Todo Item (li) * N
         ├─ TodoInput (checkbox)
         ├─ text
         └─ delete button
```

-  데이터 흐름(단방향)
    - TodoPage가 상태를 소유한다 (single source of truth)
    - TodoInput은 props로 받은 값으로 UI만 표시하고 이벤트만 위로 올린다

### 4. 의사결정(왜 이렇게 했는가)
- 상태를 TodoPage에서만 관리한 이유
    - 상태가 여러 컴포넌트로 흩어지면 추적이 어렵다.
    - 부모가 상태를 소유하고, 자식은 `props`로 받는다 -> React 기본 패턴

- LocalStorage를 붙인 이유
    - 새로고침/재접속 시 데이터 유지
    - 백엔드 없어도 서비스처럼 동작 하게 만들기 위해
    - 이후 Supabase 연동 전 단계로 적합하다 판단.

- `ul` 점(bullet) 문제를 구조 변경이 아니라 스타일로 해결한 이유
    - `<ul>`의 자식은 `<li>`가 되는 것이 `HTML`구조상 안전함
    - `ul` 정렬은 `li` 내부 레이아웃(`display: flex`등)으로 해결
